#############################################################################

 - The same variable will also control the extraction mode during timing_eco.
 - Essentially, hyper_route_opt and timing_eco steps are expected to be run with the same extraction mode.

# set_stage integration to set_qor_strategy 
###############################################################################
* In this release, the contents of the set_stage command have been integrated into the set_qor_strategy commnad. The set_stage command is no longer in use.
  Refer to the set_qor_strategy man page for details about the latest usages.

  the usage of set_stage and set_qor_strategy have been updated with new set_qor_strategy usages. Below are the details.
  - In compile.tcl, "set_qor_strategy -stage synthesis and set_stage -step synthesis" are replaced by "set_qor_strategy -stage compile_initial".
    "set_stage -step compile_place" is replaced by "set_qor_strategy -stage compile_final_place". 
  - In clock_opt_cts.tcl, "set_qor_strategy -stage pnr and set_stage -step cts" are replaced by "set_qor_strategy -stage cts"  
  - In clock_opt_opto.tcl, "set_stage -step post_cts_opto" is replaced by "set_qor_strategy -stage post_cts_opto"
  - In route_auto.tcl, "set_stage -step route" is replaced by "set_qor_strategy -stage route"
  - In route_opt.tcl, "set_stage -step post_route" is replaced by "set_qor_strategy -stage post_route"

# PBA enablement revamp 
###############################################################################
* The PBA enablement usage in set_stage and route_opt.tcl has been streamlined to improve ease of use
* In the previous release
  - In the set_stage -post_route command, PBA is enabled based on the set_qor_strategy metric value, 
    but set_stage can also explicitly enable PBA, regardless of metric, if the RM variable ENABLE_ROUTE_OPT_PBA 
    is set to true.
  - In route_opt.tcl, PBA can be turned off if the RM variable ENABLE_ROUTE_OPT_PBA is set to false,
    but the script does not enable PBA if ENABLE_ROUTE_OPT_PBA is true.
* In this release, the usage has been simplified to improve EoU
  - The set_stage -post_route command now enables PBA only based on the set_qor_strategy metric value.
    ENABLE_ROUTE_OPT_PBA is no longer impacting the set_stage command.
    Setting ENABLE_ROUTE_OPT_PBA to true will not trigger the set_stage command to enable PBA.
  - In route_opt.tcl, the script can now enable, disable, or keep current PBA setting based on the value of ENABLE_ROUTE_OPT_PBA.
    ("true" enable PBA path; "false" disables PBA, and "" (empty string) keeps existing setting untouched)
    This is done after the set_stage -post_route command amd right before the hyper_route_opt command, which allows a way to 
    override the PBA setting before hyper_route_opt.

###############################################################################
# Makefile Target consolidation
###############################################################################
* All makefiles that utilized a seperate step to create the appropriate variable file have been updated and consolidated into a single makefile target.
  For example:
  compile_pre_dft: setup init_design
        rm -f compile_pre_dft.varfile
        echo > compile_pre_dft.varfile
        echo 'set PREVIOUS_STEP $$INIT_DESIGN_BLOCK_NAME' >> compile_pre_dft.varfile
        echo 'set CURRENT_STEP $${COMPILE_BLOCK_NAME}_logic_opto' >> compile_pre_dft.varfile
        echo 'set FC_STEP_LIST "pre_initial_map initial_map logic_opto"' >> compile_pre_dft.varfile
        export RM_VARFILE=compile_pre_dft.varfile; $(FC_EXEC) $(OPTIONS) -f ./rm_fc_scripts/compile.tcl |tee -i $(LOGS_DIR)/compile_pre_dft.log

  The varfile creation now happens with the makefile flow step target.  This consolidation ensures that the varfile is always generated on demand and
  prevents the varfile from becoming stale.

# Redundant Via flow enhancement
###############################################################################
* Added support for customer redundant via insertion flow.  User can supply their own redundant via script using the variable TCL_USER_REDUNDANT_VIA_SCRIPT.

  The variable ENABLE_POST_ROUTE_OPT_REDUNDANT_VIA_INSERTION can be used to control the behavior.

# New Makefile target rm_summary
###############################################################################
* rm_summary target will run the rm_summary utility script that generates and refreshes a qor summary table.    
  To use, submit the rm_summary target in parallel with the appropriate implementation Makefile target 
  make -j -f rm_setup/Makefile route_opt rm_summary

# Enhanced compile_fusion usage model support

- FC RM now supports 4 different use modes: Monolihic, DFT Centric, Third-Party DFT, and Handoff  
  There is a new variable called FC_STEP_LIST that is used to configure the execution of each compile_fusion flow using the compile.tcl script

- FC_STEP_LIST is a task variable that configures the list of steps executed by compile_fusion 
               pre_initial_map - executes pre initial_map design setup steps 
	       initial_map - compile_fusion is executed from initial_map to initial_map 
               logic_opto - compile_fusion is executed from logic_opto to logic_opto 
               insert_dft - Assumes logic opto complete, only insert_dft is executed 
               initial_place - compile_fusion is executed from initial_place to initial_place 
               initial_drc - compile_fusion is executed from initial_drc to initial_drc 
               initial_opto - compile_fusion is executed from initial_opto to initial_opto 
               initial_opto_incremental - compile_fusion  executes  initial_opto in incremental mode 
               final_place - compile_fusion is executed from final_place to final_place 
               final_opto - compile_fusion is executed from final_opto to final_opto

	- Makefile will run the monolothic or DFT centric flow  
-	- Makefile_third_party_dft will run the third party dft flow
	- Makefile_handoff can be used to faclitate the handoff of a design from synthesis to place and route.   

- A variable file is created and used in each Makefile. It defines the appropriate FC_STEP_LIST for each makefile target.
  User is not exected to define or modify the FC_STEP_LIST variable.  The variables are automatically configured in each of the Makefiles 

- The monolithic compile_fusion flow is the traditional default FC-RM out of the box flow. Compile_fusion is run from initial_map through final_opto in a single tool session
- The DFT centric flow runs compile_fusion in three tool sessions. First compile_fusion is run fom initial map to logic opto, then inset_dft, 
  and lastly initial_place to final_opto.   

- The third party DFT flow runs compile_fusion through initial_opto. DFT insetion within FC is disabled.  Data is written out after initial_opto.
  After 3rd party DFT, init_design rebuilds the database (default is ASCII), and flow continues from incremental initial_opto to final_opto.  

- The handoff flow runs compile_fusion through initial_opto. Data is written out after initial_opto and can be handed off via ASCII or NDM.  This data can then be used to continue 
  from final_place to final_opto.

# Automatic route_detail -incremental after (hyper_)route_opt
###############################################################################
- Adds a new step to automatically run "route_detail -incremental true -initial_drc_from_input true" after the core command in route_opt.tcl (core command : hyper_route_opt) and in endpoint_opt.tcl 
  (core command : targeted_ep_ropt_pba_ccd) to help work on routing DRC.
  By default, the command automatically triggers when the following conditions are met :  
  (1) DRC increases after the core command (2) DRC increase percentage is larger than a specified threshold (defaull is 10%) (3) DRC before core command is less than a specified threshold (default is 10000) 
  (4) DRC after the core command is at least larger than a specified threshold (default is 50)

- New Tcl variables are added to design_setup.tcl to allow user configuration of the feature or turning the feature off
  - INCR_ROUTE_DETAIL_MODE is the main switch and default is "auto" to trigger this feature based on the said conditions. 
    Set it to false to always disable and true to unconditionally enable it.
  - INCR_ROUTE_DETAIL_MAX_ITERATIONS is default null. This is an optional variable to control the -max_number_iterations of the route_detail command with an iteration number differnt than tool default. 

  The following variables are only takes effect when INCR_ROUTE_DETAIL_MODE is auto :
  - INCR_ROUTE_DETAIL_DRC_INCREASE_THRESHOLD_MIN is default "0.1" (i.e. 10%). If DRC increase (i.e. DRC after core command - DRC before core command) divided by DRC before core command is smaller than 
    this value, the feature won't trigger. A larger value requires the DRC increase to be much larger in order to trigger route_detail.
  - INCR_ROUTE_DETAIL_DRC_THRESHOLD_MAX is default 10000. The feature won't trigger if routing DRC before the core command is larger than this value. This avoids the feature to work on unrouteable designs.
  - INCR_ROUTE_DETAIL_DRC_THRESHOLD_MIN is default 50. The feature won't trigger if routing DRC after the core command is smaller than this value. This avoids the feature to work on designs with small DRCs.

# Added new commands check_clock_transition and restore_clock_transition
###############################################################################
* The check_clock_transition command detects and applies a clock max transition constraint on the fastest clock(s) per active scenario to improve OOTB CTS QoR
  The max transition constraint is based on a percentage of the period of the fastest clock.  That percentage is specified with the -threshod option.  
  A default threshold of 15% will be used with check_clock_transition in clock_opt_cts.tcl    

* If the incoming clock max transition constraint is greater than the threshold max transition, the command will issue an informative message. 
  When -apply_max_transition is specified, the derived max transition constraint will be applied to the clock if the constraint is tighter 
  than the incoming clock max transition constraint.
  If incoming clock max transition constraint is tighter than the threshold, clock max transition is maintained.  

* If the clock transition was updated by the check_clock_transition command, it will only be applied for clock_opt build_clock.  The command restore_clock_transition will 
  restore the clock max transition back to the incoming clock max_transition constraint after the clocks have been synthesized.

###############################################################################
# FC RM will only support the in-compile DFT synthesis flow
###############################################################################
* Starting from R-2020.09-SP6 release, Fusion Compiler FE will only support the "in-compile" DFT synthesis flow.
  As a result, previous RM configuration settings to enable "pre-compile" DFT flow will no longer be available.

# New reporting command check_stage_settings
###############################################################################
* A new command check_stage_settings is available in the RM scripts.  The check_stage_settings command will 
  be run just before the the major stage command is executed.  The command compares current app option settings 
  against set_technology, set_qor_strategy, set_stage, and select tool default settings which can impact runtime or ppa.
  check_stage_settings will output only app options that have current settings that are different from the recommended setting.
  The command also has the ability to reapply set_technology, set_qor_strategy, and set_stage if a difference is detected.  This
  can be enabled with the variable RESET_CHECK_STAGE_SETTINGS.  The variable is defined in fc_setup.tcl.

###############################################################################
# Scenario power analysis configuration for set_qor_strategy metrics 
###############################################################################

* When setting the SET_QOR_STRATEGY_METRIC variable to timing or leakage_power in
  fc_setup.tcl the clock_opt_cts.tcl, clock_opt_opto.tcl, and route_opt.tcl scripts
  manages the power analysis of active scenarios to direct optimization towards
  the specified metric. 
  
  - When set to timing, no power optimization is directed to the tool in preroute. 
  If there is an active scenario with the leakage_power and dynamic_power analysis
  set to true, the RM scripts set the analysis to false during the clock_opt 
  command. The scenario status is set back after optimization for reporting.
  This directs the commands to not take leakage or dynamic power into consideration
  and not do any power optimization.
  - When set to leakage_power, only leakage power optimization is directed to the
  tool. If there is an active scenario with dynamic_power analysis enabled then the 
  RM scripts turn the dynamic_power analysis false for the clock_opt and route_opt 
  commands. The analysis will be turned back to true after optimization for reporting.
  This directs optimization to only do leakage power optimization and not take
  dynamic power into consideration and cause the tool to go into total power 
  optimization.

# Changed Setup file RM variables Organization
###############################################################################

* The RM variables in design_setup.tcl and fc_setup.tcl have been reorgnized to 
  focus the two files under the following guidance.
 
  - design_setup.tcl 
    A file for design specific setup variables. This includes design input files
    such as RTL, Floorplan, UPF, and reference libraries. It is also a file for design 
    specific plugin file definitions for user defined to be sourced before or after 
    major command steps. System related variables for block names, reporting, and 
    the search_path are also defined in this file.

  - fc_setup.tcl
    A file for special features and control of the RM flow. This file defines variables
    to control features and flows in the RM for different steps. For instance, defining the
    Multisource Clock Tree Synthesis flow or the Path Based Analysis Optimization for 
    post-route. Variables defined here enables these features through a combination of 
    RM script flow changes and/or the set_stage mega switch.

# Removed settings.<step>.tcl scripts
###############################################################################

* The settings.<step>.tcl files have been removed and consolidated into the main command
  scripts and mega switches. The contents of the settings.<step>.tcl files typically applied
  different app_option settings based on fc_setup.tcl variable definitions or sourced plugin
  files to define things like non-default routing rules. The sourcing of plugin files moved
  to the main scripts and the settings applications have been put into the set_stage mega
  switch. Based on the feature variables defined in the fc_setup.tcl set_stage performs 
  different app_option setup for the specified step.
